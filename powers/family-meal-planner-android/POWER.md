# Family Meal Planner Android Development Guide

## Overview

This power provides comprehensive guidelines for developing the Family Meal Planner Android application. The app is built using modern Android development practices with Jetpack Compose, following MVVM with Clean Architecture principles. It helps families organize recipes, manage pantry inventory, create grocery lists, and plan meals.

The app features 50+ bundled recipes across multiple cuisines, smart ingredient matching, vibe-based meal recommendations, voice input for ingredients, and comprehensive allergen management. It supports multi-language localization (English, Russian, Romanian) and offers extensive customization options.

## Architecture Overview

### Clean Architecture with MVVM

The project follows a three-layer architecture:

**Data Layer** (`data/`)
- Local data sources (Room database, DataStore preferences)
- Remote data sources (API services)
- Repository implementations
- Data models and entities

**Domain Layer** (`domain/`)
- Business logic and use cases
- Domain models (pure Kotlin classes)
- Repository interfaces
- Domain utilities

**Presentation Layer** (`ui/`)
- Jetpack Compose screens and components
- ViewModels (state management)
- Navigation setup
- UI theme and utilities

### Dependency Injection

The app uses Hilt for dependency injection with the following modules:

- `AppModule` - Application-level dependencies
- `DatabaseModule` - Room database and DAOs
- `NetworkModule` - Ktor HTTP client and API services
- `RepositoryModule` - Repository implementations
- `ImageModule` - Coil image loading configuration

## Room Database Best Practices

### Schema Management

**Development Phase**
- Multiple schema versions exist in `app/schemas/` directory (versions 1-13)
- Current development version: 13
- Schema files are auto-generated by Room during compilation

**Production Release**
- Final app will launch with a single schema version
- All development schemas will be consolidated
- Migration paths will be removed for clean production database

### Entity Design

**Naming Conventions**
```kotlin
@Entity(tableName = "meals")
data class MealEntity(
    @PrimaryKey val id: String,
    @ColumnInfo(name = "meal_type") val mealType: String,
    @ColumnInfo(name = "created_at") val createdAt: Long
)
```

**Key Principles**
- Use explicit table names with `tableName` parameter
- Use snake_case for column names with `@ColumnInfo`
- Use camelCase for Kotlin property names
- Always include `createdAt` and `updatedAt` timestamps
- Use String UUIDs for primary keys

### Foreign Keys and Relationships

```kotlin
// Define foreign keys with proper cascade behavior
@Entity(
    tableName = "meal_ingredients",
    foreignKeys = [
        ForeignKey(
            entity = MealEntity::class,
            parentColumns = ["id"],
            childColumns = ["meal_id"],
            onDelete = ForeignKey.CASCADE
        ),
        ForeignKey(
            entity = IngredientEntity::class,
            parentColumns = ["id"],
            childColumns = ["ingredient_id"],
            onDelete = ForeignKey.RESTRICT
        )
    ],
    indices = [
        Index("meal_id"),
        Index("ingredient_id")
    ]
)
```

**Cascade Rules**
- `CASCADE` - Delete child records when parent is deleted (meal_ingredients → meals)
- `RESTRICT` - Prevent deletion if child records exist (ingredients referenced by meals)
- Always create indices on foreign key columns for performance

### Database Migrations

**During Development**
- Increment version number in `@Database` annotation
- Room auto-generates schema JSON files
- Use destructive migration for development: `.fallbackToDestructiveMigration()`

**For Production**
- Write explicit migration paths using `Migration` class
- Test migrations thoroughly with existing data
- Never use destructive migration in production

```kotlin
val MIGRATION_12_13 = object : Migration(12, 13) {
    override fun migrate(database: SupportSQLiteDatabase) {
        database.execSQL(
            "ALTER TABLE meal_plans ADD COLUMN ingredient_substitutions TEXT NOT NULL DEFAULT '{}'"
        )
    }
}
```

### DAO Best Practices

```kotlin
@Dao
interface MealDao {
    // Use Flow for reactive queries

}
```

## Recipe Data Handling

### JSON Structure (CRITICAL)

**The recipe JSON structure MUST be preserved exactly as defined. Never use scripts or automated tools to modify recipe files.**

**Required Recipe JSON Format**
```json
{
  "name": "Recipe Name",
  "instructions": "Detailed step-by-step instructions...",
  "simpleInstructions": "Simplified instructions...",
  "prepTimeMinutes": 20,
  "cookTimeMinutes": 35,
  "servings": 2,
  "mealType": "BREAKFAST|LUNCH|DINNER|SNACK|DESSERT",
  "dishCategory": "MAIN_COURSE|SIDE_DISH|APPETIZER|SOUP|SALAD|DESSERT|BEVERAGE",
  "ingredients": [
    {
      "name": "Ingredient Name",
      "quantity": 100,
      "unit": "g|ml|pcs|tsp|tbsp|cup",
      "isStarIngredient": true|false
    }
  ],
  "sourceUrl": "https://...",
  "cuisine": "Asian|Italian|Mexican|...",
  "id": "unique_recipe_id",
  "imageUrl": "recipes/images/category/recipe_name.jpg"
}
```

### Recipe Modification Guidelines

**CRITICAL RULES**

1. **Manual Assessment Required**: When fixing recipes, manually review each file to identify what needs correction
2. **No Automated Scripts**: Never use PowerShell, bash scripts, or automated tools to modify recipe JSON files
3. **Preserve Structure**: Maintain exact JSON structure, field names, and data types
4. **One File at a Time**: Edit recipe files individually, never batch process
5. **Validate After Changes**: Verify JSON syntax and structure after each edit

**Common Recipe Fixes**

When asked to fix recipes, follow this process:

1. **Identify the Issue**: Read the recipe file to understand what's wrong
   - Missing fields
   - Incorrect data types
   - Invalid enum values
   - Malformed JSON

2. **Manual Correction**: Edit the specific fields that need fixing
   - Use proper JSON syntax
   - Ensure all required fields are present
   - Validate enum values (mealType, dishCategory, unit)
   - Check ingredient structure

3. **Verify Changes**: After editing, confirm:
   - JSON is valid (no syntax errors)
   - All required fields present
   - Data types match schema
   - Ingredient quantities are reasonable

**Example: Fixing a Recipe**

```kotlin
// WRONG: Using scripts to batch modify
// ❌ Never do this
executeBash("find recipes -name '*.json' -exec sed -i 's/old/new/g' {} \\;")

// CORRECT: Manual file-by-file review and editing
// ✅ Always do this
// 1. Read the specific recipe file
// 2. Identify the issue (e.g., missing mealType)
// 3. Edit the file to add: "mealType": "DINNER"
// 4. Verify JSON is valid
// 5. Move to next recipe if needed
```

## Common Patterns

### Repository Pattern

```kotlin
interface RecipeRepository {
    fun getAllRecipes(): Flow<List<Recipe>>
    suspend fun getRecipeById(id: String): Recipe?
    suspend fun insertRecipe(recipe: Recipe)
    suspend fun deleteRecipe(id: String)
}

class RecipeRepositoryImpl @Inject constructor(
    private val mealDao: MealDao
) : RecipeRepository {
    override fun getAllRecipes(): Flow<List<Recipe>> =
        mealDao.getAllMeals().map { entities ->
            entities.map { it.toDomainModel() }
        }
    
    override suspend fun getRecipeById(id: String): Recipe? =
        mealDao.getMealById(id).firstOrNull()?.toDomainModel()
    
    override suspend fun insertRecipe(recipe: Recipe) {
        mealDao.insertMeal(recipe.toEntity())
    }
    
    override suspend fun deleteRecipe(id: String) {
        mealDao.deleteMeal(id)
    }
}
```

### Use Case Pattern

```kotlin
class GetRecipeSuggestionsUseCase @Inject constructor(
    private val recipeRepository: RecipeRepository,
    private val ingredientRepository: IngredientRepository
) {
    suspend operator fun invoke(
        mealType: MealType? = null,
        dishCategory: DishCategory? = null
    ): Result<List<RecipeSuggestion>> = try {
        val availableIngredients = ingredientRepository.getAvailableIngredients().first()
        val recipes = recipeRepository.getAllRecipes().first()
        
        val suggestions = recipes
            .filter { recipe ->
                (mealType == null || recipe.mealType == mealType) &&
                (dishCategory == null || recipe.dishCategory == dishCategory)
            }
            .map { recipe ->
                val matchPercentage = calculateMatchPercentage(
                    available = availableIngredients,
                    required = recipe.ingredients
                )
                RecipeSuggestion(recipe, matchPercentage)
            }
            .sortedByDescending { it.matchPercentage }
        
        Result.success(suggestions)
    } catch (e: Exception) {
        Result.failure(e)
    }
}
```

### ViewModel Pattern

```kotlin
@HiltViewModel
class RecipeListViewModel @Inject constructor(
    private val getRecipeSuggestionsUseCase: GetRecipeSuggestionsUseCase
) : ViewModel() {
    
    private val _uiState = MutableStateFlow<RecipeListUiState>(RecipeListUiState.Loading)
    val uiState: StateFlow<RecipeListUiState> = _uiState.asStateFlow()
    
    fun loadRecipes(mealType: MealType? = null) {
        viewModelScope.launch {
            _uiState.value = RecipeListUiState.Loading
            getRecipeSuggestionsUseCase(mealType = mealType)
                .onSuccess { suggestions ->
                    _uiState.value = RecipeListUiState.Success(suggestions)
                }
                .onFailure { error ->
                    _uiState.value = RecipeListUiState.Error(error.message ?: "Unknown error")
                }
        }
    }
}

sealed class RecipeListUiState {
    object Loading : RecipeListUiState()
    data class Success(val suggestions: List<RecipeSuggestion>) : RecipeListUiState()
    data class Error(val message: String) : RecipeListUiState()
}
```

## Troubleshooting

### Common Issues

**Room Schema Mismatch**
- **Problem**: "Room cannot verify the data integrity"
- **Cause**: Database schema changed without migration
- **Solution**: Increment version number, add migration, or use `.fallbackToDestructiveMigration()` in development

**Compose Recomposition Issues**
- **Problem**: UI not updating when state changes
- **Cause**: State not properly observed or remembered
- **Solution**: Use `collectAsState()` for Flow, ensure state is hoisted correctly

**Hilt Injection Failures**
- **Problem**: "Cannot find symbol" or injection errors
- **Cause**: Missing `@HiltAndroidApp`, `@AndroidEntryPoint`, or module bindings
- **Solution**: Verify annotations, check module provides/binds methods

**Image Loading Failures**
- **Problem**: Images not displaying
- **Cause**: Incorrect path, missing permissions, or network issues
- **Solution**: Verify image paths, check internet permission, use placeholder images

### Debugging Tips

**Database Inspection**
- Use Android Studio Database Inspector
- Export database for manual inspection

**Compose Layout Inspector**

- Verify state flow
